淺談Java的常數池

Answer

這是什麼巫術 ?

首先，在Java中的==有兩種意涵：

對於基本資料型態，比對兩者的數值
對於物件，則是比對兩者存放的位置

String是個物件，所以對String使用==，我們其實是在比較物件的記憶體位置，這就解決了最後一個問題，s3與s10都是經過new配置而成，
實際上new傳回了兩個String，只不過他們的字串值都叫做Hello而已，比對兩者的位置當然會得出false。

如果明白了最後一題在搞什麼花樣，問題來了，為什麼第一題是true？

常數池

讓我們從答案往回推，既然第一題是true，也就代表s1與s2所指向的是同一個物件，而為何會指向同一個物件呢？會不會是Java存在一個特殊空間，把編譯時期就已經確定的常數(constant)包進.class，執行時再從這個空間統一拿出？

正確答案，這個特殊空間稱為常數池(constant pool)，這玩意兒在JVM中相當重要，它存放兩種東西：

  • 字面量(Literal)
    包含了常數以及既定字串。

  • Symbolic References
    呃…我不知道他中文叫什麼，姑且稱為符號參考吧，主要是類別和介面的Fully qualified name，欄位和方法的名稱及修飾字。

除了String，Java對double及float以外的基本類別在區間-128~127都實作了常數池，也就是說

關於這點，我們也可從Integer類別中valueOf的實作得到證實：

P.S
我會提到valueOf是因為Integer i1 = 1其實是個編譯器蜜糖，編譯器會將上述程式碼展開成Integer i1 = valueOf(1)，對於這部分更深入的細節，可以餵食Google「Java裝箱拆箱」。

一般來說，常數池是這麼運作的(以string及題目為例)：

  1. 當載入class時，JVM會尋找所有程式中的字串字面量(string literals)
  2. 首先，他會找到字串s1的字面量Hello，並產生一個實例放到記憶體中，嚴格來說是放置字串常數池(string constant pool)的記憶體區。
  3. 接下來，JVM會複製一份Hello，這很重要，其避免了對s1的操作影響到常數池的維護。
  4. 之後JVM找到s2的字面量Hello，不過它發現記憶體中已經有一個生成的Hello，所以JVM會透過常量池的指針將s2指向已經生成好的Hello(s1所持有的)，而非再產生一個Hello。

由此我們也能發現常量池的優點，它達到了共享資源的效果，避免資源頻繁產生或釋放造成不必要的浪費。

關於第二題 (s1==s3) 為false，雖然兩者都是存放在heap，但是因為new總是產生出新的物件，其跟s1接收到的Hello自然是不同的，我們能這麼理解。

heap
“Hello” <- s1,s2

在 s3 = new String("Hello");

heap
“Hello” <- s3
“Hello” <- s1,s2

但是要注意，new 仍舊會觸發常數池存取，如果今天s3 = new String("Hello");是第一行，常數池的處理會變成這樣：

  1. JVM發現了字串常量Hello，產生一份放在字串常數池中
  2. 不過 s3 使用的是 new ，所以其不引用常數池，而是在heap生成一個新的String物件

簡單來說，= 與 new 都會在常數池產生物件，不過 = 會引用常數池在heap中生成的複製品，new則是另外生成新的物件。

關於第三題 (s1==s4) 為true，是Java的優化功能之一，編譯器會將"Hel"+"lo"當作"Hello"，因此道理和(s1==s2)相同。

關於第四題，近似於第二題的概念，具體來說String s5 = "Hel" + new String("lo");做了三件事：

  1. 常數池存放 “Hel” 及 “lo”
  2. heap 產生 “Hel” 與 “lo”
  3. 使用String方法.append()合併”Hel”與”lo”，傳回一個新字串給s5

第五題則是和第三題的類比，會回傳false可不是因為它是拆成He與llo，而是跟先前提到的優化有關，它只會進行常數的優化，這是因為編譯器不知道我們在String s9 = s7+s8;這行前，會不會對s7與s8的值進行修改，若貿然的假設s7仍是He或s8仍是llo，反而可能造成預期外的錯誤，這風險可不是編譯器能承擔的，所以自然也不會處理這個部分。
